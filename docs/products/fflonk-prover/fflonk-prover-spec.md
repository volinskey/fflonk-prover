## Overview

`fflonk-prover` is a native Rust FFLONK prover that consumes Circom-generated R1CS constraints and produces proofs verifiable by snarkjs-generated Solidity verifiers. It exists because snarkjs (the standard JavaScript prover) is too slow for production use ($3/proof), while the only fast native prover (rapidsnark) only supports Groth16 which requires a per-circuit Phase 2 ceremony. This prover fills the gap: native speed + FFLONK proof system (no Phase 2 ceremony needed).

PLONK support may be added in a future version but is out of scope for v0.1.0.

The prover is **outside the trust boundary** by design — it cannot forge proofs regardless of bugs. It can only produce valid proofs (verifier accepts), invalid proofs (verifier rejects), or crash. The security of the system lives in the circuit (audited), the verifier (deterministic), and the SRS (Hermez ppot, 86+ contributors). The prover is a performance optimization, not a security component.

## Interfaces & Mediums

**CLI binary + Rust library.** Two consumption modes:

1. **CLI** — standalone binary invoked from shell scripts or orchestration. Takes proving key + witness file as input, outputs proof JSON. Drop-in replacement for the proving step in existing Circom pipelines.
2. **Library (crate)** — Rust crate importable by other Rust programs. The kysigned operator service will use this mode to generate proofs inline without shelling out.

No web interface, no API, no mobile. This is developer/operator infrastructure.

## Features & Requirements

### F1 — FFLONK proof generation

**F1.1** The prover accepts a snarkjs-generated FFLONK proving key (`.zkey` file) and a Circom-generated witness (`.wtns` file) as inputs, and produces a proof file as output.

**F1.2** The output proof is byte-compatible with the snarkjs-generated FFLONK Solidity verifier. Specifically: the proof passes `verifyProof()` on the Solidity contract that `snarkjs fflonk setup` + `snarkjs zkey export solidityverifier` produces from the same R1CS + ptau.

**F1.3** The Fiat-Shamir transcript construction (challenge derivation) matches snarkjs's FFLONK implementation exactly — same field-to-bytes encoding, same point serialization, same hashing domain, same ordering. Incompatible transcripts produce proofs the verifier rejects (safe failure, not security failure).

**F1.4** The prover supports BN254 (alt_bn128) curve, which is the curve used by Hermez ppot and all Circom-ecosystem tooling.

**F1.5** The prover uses the Hermez Perpetual Powers of Tau Phase 1 SRS. No Phase 2 ceremony input is needed (FFLONK is universal — Phase 1 only).

### F2 — Input/output formats

**F2.1** Input proving key: snarkjs FFLONK `.zkey` binary format. The prover reads this file — it does NOT generate it. Setup is done by snarkjs (same codebase as verifier, keeps the transpiler in trusted hands).

**F2.2** Input witness: Circom `.wtns` binary format, as generated by the circom WASM witness calculator or any compatible witness generator.

**F2.3** Output proof: JSON format matching snarkjs's FFLONK proof output structure. The JSON fields and encoding must match exactly so that `snarkjs zkey export soliditycalldata` (or equivalent) produces valid calldata for the on-chain verifier.

**F2.4** Output public signals: JSON array of field elements matching snarkjs's `public.json` format.

### F3 — Performance

**F3.1** The prover uses all available CPU cores for parallelizable operations (MSM, NTT/FFT). The degree of parallelism is configurable via a `--threads` CLI flag or library API parameter. Default: number of physical cores.

**F3.2** For a 4.7M-constraint Circom R1CS (the `kysigned-approval.circom` circuit), the prover completes proof generation in under 60 seconds on a 16-vCPU machine (r5.4xlarge equivalent). Target: under 20 seconds (comparable to rapidsnark's 13.49s Groth16 time on the same circuit and hardware).

**F3.3** Peak memory usage for a 4.7M-constraint circuit is under 8 GB. Target: under 4 GB (comparable to rapidsnark's 3.57 GB).

### F4 — CLI interface

**F4.1** The CLI binary is named `fflonk-prover` and supports the following subcommands:

- `fflonk-prover prove <zkey> <witness> <proof-out> <public-out>` — generate a proof
- `fflonk-prover verify <verification-key> <public> <proof>` — verify a proof locally (for testing; not the on-chain verifier)
- `fflonk-prover info <zkey>` — display proving key metadata (proof system, constraint count, curve)
- `fflonk-prover --version` — print version

**F4.2** Exit codes: 0 = success, 1 = invalid input / proof generation failed, 2 = verification failed.

**F4.3** The `prove` subcommand prints timing information to stderr: total proving time, breakdown by phase (FFT, MSM, polynomial evaluation, commitment), peak memory.

### F5 — Library API

**F5.1** The Rust crate exposes a `prove(zkey: &[u8], witness: &[u8]) -> Result<(Proof, PublicSignals), ProverError>` function (or equivalent typed API).

**F5.2** The library is `no_std`-compatible where feasible, but `std` is the primary target. Async is not required — proving is CPU-bound synchronous work.

**F5.3** The library re-exports types for `Proof` and `PublicSignals` that serialize to the snarkjs-compatible JSON format via `serde`.

### F6 — Correctness validation

**F6.1** The prover ships with an integration test that: (a) takes a known R1CS + ptau, (b) runs snarkjs to generate the FFLONK proving key + verification key + Solidity verifier, (c) generates a witness from a known input, (d) generates a proof using the Rust prover, (e) verifies the proof using snarkjs's JavaScript verifier, confirming byte-compatibility.

**F6.2** The integration test also deploys the Solidity verifier to a local Anvil/Hardhat node and submits the proof on-chain, confirming the proof passes the on-chain verifier.

**F6.3** A reference test vector set is committed to the repo: known (R1CS, witness, proving key, proof, public signals) tuples for FFLONK, generated by snarkjs. The Rust prover must produce proofs that pass the same verification key for the same witness.

### F7 — Licensing and dependencies

**F7.1** The prover is licensed MIT. All dependencies must be MIT or Apache-2.0 (or dual MIT/Apache-2.0). No GPL, LGPL, MPL, or other copyleft dependencies.

**F7.2** Dependencies are pinned via `Cargo.lock` (committed to repo) for reproducible builds.

## Constraints & Dependencies

### Hard constraints

1. **Byte-compatibility with snarkjs verifiers.** The prover's output must pass snarkjs-generated verifiers without modification. This is the primary correctness constraint. The Fiat-Shamir transcript (challenge derivation), point serialization, and field element encoding must match snarkjs exactly.

2. **No setup responsibility.** The prover does NOT perform R1CS-to-FFLONK transpilation, does NOT generate proving keys or verification keys, does NOT generate Solidity verifiers. All setup is done by snarkjs. The prover only reads the resulting proving key and generates proofs. This keeps the transpiler (which IS in the trust boundary) in snarkjs's trusted codebase.

3. **MIT-only dependency tree.** No copyleft. This is a public open-source project intended for broad adoption.

4. **BN254 curve only.** No other curves needed. BN254 is the standard for Circom/snarkjs/EVM-precompile ecosystem.

## Open Questions

1. **Should the prover also support witness generation?** Currently, witness generation is done by circom's WASM code (via `ark-circom`). Embedding witness generation in the prover binary would make it a single-binary pipeline (`fflonk-prover prove-from-input <zkey> <input.json> <wasm> ...`). Trade-off: larger scope vs. simpler deployment.

2. **Should we vendor snarkjs test fixtures or generate them in CI?** Vendoring is simpler but creates staleness risk. Generating in CI requires Node.js + snarkjs in the CI environment.

3. **Target the `kysigned-approval.circom` circuit specifically, or be circuit-agnostic?** Being circuit-agnostic is more useful but harder to test. The kysigned circuit (4.7M constraints) is the primary use case.

4. **Proving key format: read snarkjs `.zkey` directly, or define our own optimized format?** Reading `.zkey` directly is simpler and ensures compatibility. A custom format could be faster to load but adds a conversion step.
